Host an API Endpoint on a Subdomain with AWS Lambda

Use an is-pro.dev subdomain to expose serverless API endpoints built with AWS Lambda and API Gateway, with proper CORS configuration and custom domain routing.

Overview

AWS Lambda + API Gateway lets you build serverless APIs without managing servers. By routing through an is-pro.dev subdomain, you get a clean URL with Cloudflare's edge caching for your API responses.

Prerequisites

  • An AWS account with Lambda and API Gateway access
  • A serverless function written in Node.js, Python, or Go
  • An is-pro.dev subdomain registered at dash.is-pro.dev

Step 1: Deploy Your Lambda Function

Write your handler function in the AWS Lambda console or deploy via the AWS CLI. Example Node.js handler: exports.handler = async (event) => ({ statusCode: 200, body: JSON.stringify({ message: "Hello from API" }) }). Set up API Gateway as a trigger — create a REST or HTTP API. Note the "Invoke URL" from API Gateway (e.g. https://abc123.execute-api.us-east-1.amazonaws.com).

Step 2: Configure Custom Domain in API Gateway

In API Gateway → Custom Domain Names, create a new domain name. Enter your is-pro.dev subdomain. Upload or create a TLS certificate via AWS Certificate Manager (ACM) in us-east-1. AWS issues the certificate automatically after DNS verification. Map the API Gateway stage (e.g. prod) to a base path (/).

Step 3: Configure DNS on is-pro.dev

Add a CNAME record on the is-pro.dev dashboard pointing your subdomain to the API Gateway distribution domain (from the Custom Domain Names page). Enable Cloudflare proxy (orange cloud) for free SSL — set SSL/TLS mode to "Full (Strict)" to ensure encryption reaches API Gateway.

Step 4: Set Up CORS for Frontend Access

If your API is called from a browser-based app on a different subdomain, add CORS headers in your Lambda response: Access-Control-Allow-Origin: https://app.yourname.is-pro.dev, Access-Control-Allow-Methods: GET, POST, OPTIONS. For OPTIONS preflight requests, return a 200 response with the same CORS headers.

Best Practices

  • Use API Gateway usage plans and API keys to control access if needed
  • Enable Cloudflare caching for GET endpoints by setting Cache-Control headers in your Lambda
  • Log API requests with CloudWatch and set up error rate alerts
  • Use Lambda environment variables (encrypted) for secrets like database credentials

FAQ

Does Cloudflare proxy work with API Gateway?

Yes — set Cloudflare SSL/TLS to "Full (Strict)" and ensure the ACM certificate covers your custom domain. API Gateway handles the origin-side SSL.

How do I handle API versioning?

Use path-based versioning in API Gateway: map /v1 to one stage and /v2 to another. Your is-pro.dev subdomain stays the same — clients access api.yourname.is-pro.dev/v1/endpoint.
